---
title: "Error format"
description: "Stable error shape across every Maroa endpoint."
---

Every non-2xx response from the Maroa API uses this shape:

```json
{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "businessId required",
    "details": null,
    "timestamp": "2026-05-20T14:23:00.123Z"
  }
}
```

| Field | Type | Notes |
|---|---|---|
| `code` | string | Stable machine-friendly identifier. Don't parse `message`. |
| `message` | string | Human-readable. Safe to surface to end-users. |
| `details` | object \| null | Optional extra context. Field-specific keys vary by error. |
| `timestamp` | ISO 8601 | Server-side. Use this when filing bug reports. |

## Common codes

| Code | Status | Meaning |
|---|---|---|
| `VALIDATION_ERROR` | 400 | Request body / params didn't match the expected shape |
| `UNAUTHORIZED` | 401 | Bearer JWT missing or expired |
| `AUTH_REQUIRED` | 401 | Endpoint needs auth; legacy fallback disabled |
| `FORBIDDEN_OWNERSHIP` | 403 | Token user doesn't own the requested resource |
| `FORBIDDEN` | 403 | Generic authorization failure |
| `NOT_FOUND` | 404 | Resource doesn't exist OR caller lacks access (we don't distinguish) |
| `RATE_LIMITED` | 429 | Plan-tier or per-IP rate limit hit |
| `COST_CAP_REACHED` | 402 | Monthly LLM budget exhausted; upgrade or wait for cycle reset |
| `IDEMPOTENCY_KEY_CONFLICT` | 409 | Same key used with different body |
| `IDEMPOTENCY_KEY_IN_FLIGHT` | 409 | Concurrent request with this key still running |
| `ALREADY_APPROVED` / `ALREADY_REJECTED` | 409 | Resource already in terminal state |
| `ANTHROPIC_CIRCUIT_OPEN` | 503 | Upstream LLM circuit breaker tripped; retry after `cooldownMs` |
| `SERVICE_UNAVAILABLE` | 503 | Backend dependency unhealthy (Supabase, etc.) |
| `INTERNAL_ERROR` | 500 | Something we didn't anticipate. The timestamp is the key for our logs. |

## What to do on 5xx

Retry with exponential backoff (start 200ms, cap 20s, full jitter). After
3 retries, surface the error to the user and move on. Don't loop forever.

If `Retry-After` is present, honor it.

## Reporting bugs

If you see a 500 you can't explain, paste the `error.timestamp` value
into a GitHub issue. We index every error by timestamp; we can find
your exact request in production logs.
